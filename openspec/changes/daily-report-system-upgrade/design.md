## Context

当前日报系统（`report_task.py`）有两种路径：规则引擎（`get_daily_report`）用硬编码模板从聚合计数器产中文文本；DeepSeek AI（`get_deepseek_daily_report`）接收相同聚合数字通过 `response_format: json_object` 生成三段内容（summary/key_findings/recommendations）。两次调用均不持久化，前端刷新即丢失。周报/月报提供 `GET /weekly/` / `GET /monthly/` 端点，同机制无持久化。

本次升级目标：日报持久化 + 定时自动生成 + AI 结构化洞察 + API Key 管理 + 周报快照。

## Goals / Non-Goals

**Goals:**
- 日报数据持久化到 `daily_reports` 表，按日期 upsert
- 北京时间 17:00 自动生成当日日报（范围 00:00~17:00 CST），次日凌晨补充昨日余量
- 统计层无需 AI，规则引擎产出可直接展示的图表数据
- 洞察层由 DeepSeek 按分步 workflow 生成结构化 JSON（预测数据、风险分布），前端渲染图表
- API Key 通过 `.env` + 用户覆盖值管理
- 前端「立即生成」按钮支持即时触发（用于演示）
- 周报/月报持久化到 `weekly_reports` 表，每周一凌晨快照生成

**Non-Goals:**
- 不实现自然语言问答/交互式分析
- 不替换现有周报/月报聚合统计逻辑（只加持久化）
- 不引入新的外部 AI 提供商（仅 DeepSeek）
- 不改造告警引擎或围栏模块

## Decisions

### 1. 双层报告模型
统计层和洞察层解耦。统计层永远可用（无 API key 也能产），洞察层需要 DeepSeek key。数据库存两列：`stats_json`（规则引擎产出）和 `insights_json`（AI 产出，可为 NULL）。

**Rationale**: 即使 AI 不可用，日报仍有基本统计价值。重写洞察不影响统计。

**Alternatives**: 单 JSON 混合存储 → 拒绝，因为重写 AI 洞察时不应丢失已有统计数据。

### 2. 时间边界：北京时间 17:00 割接
日报时间范围 = 当日 00:00~17:00 CST（前一天 17:00~23:59 在次日凌晨补上）。本质是"从昨天 17:00 到今天 17:00 的 24 小时"滑动窗口。

**Rationale**: 17:00 是典型值班交接/巡检节点，符合安保运营节律。

**Alternatives**: 0:00 生成 → 拒绝，因为当天还没有足够数据；滚动 24h → 生成时间不固定，前端展示混乱。

### 3. AI 分步工作流
不分段混在一起让 AI 一次生成全部，而是三步：
1. 数据概览（`POST` 全量结构化事件 → AI 分析模式）
2. 趋势与发现（基于步骤 1 的输出 + 时序数据）
3. 建议与预测（基于前两步输出，产出预测曲线数据）

每步产出独立 JSON 属性，失败时前一步结果仍可用。

**Rationale**: 单步大 prompt 产出的结构化质量差。分步让每步专注一件事，降低幻觉。

**Alternatives**: 高 token 单次调用 → 拒绝，因为结构化质量不可控。Agent 化（工具调用） → 过度设计，日报场景不需要。

### 6. 时区策略：后端 UTC + 前端 CST
数据库所有时间戳存 UTC（与项目其他模型一致）。调度器内部使用 `Asia/Shanghai` 时区计算边界。API 响应中的 `stats_json.time_range_start/end` 带 `+08:00` 时区信息。前端从后端拿到的 UTC 时间在显示时转北京时间，页面底部标注"所有时间均为北京时间 (UTC+8)"。

**Rationale**: 后端统一 UTC 避免时区 bug；前端本地化显示让运营人员看到当地时间。调度器绑定时区是为了 17:00 CST 精确触发。

**Alternatives**: 全栈统一 CST → 拒绝。其他模型已用 UTC，混用会导致查询不一致。

### 4. API Key 三级回退
```
用户手动输入（前端保存到 localStorage + 服务端 session）
  → 如果无输入 → .env DEEPSEEK_API_KEY
    → 如果无 key → 统计层可用，洞察层灰显"请输入 API Key"
```

**Rationale**: 灵活性 + 安全性。用户可临时用自己的 key 测试，定时任务自动用 env key。

### 5. 周报持久化快照
`weekly_reports` 表：`week_start(unique) + stats_json + generated_at`。每周一凌晨 00:00 CST 自动生成前一周的统计快照（无 AI）。前端 `/reports/weekly/` GET 优先读持久化数据，无数据时回退实时计算。

## Risks / Trade-offs

- **[风险]** 同日重新生成覆盖旧报告 → 旧数据不可恢复 → **缓解**: 前端弹确认对话框，后端记录 `generated_at` 和 `regenerated_count`
- **[风险]** AI 幻觉产出不准确预测 → **缓解**: 前端标注"AI 生成，仅供参考"，预测数据附 confidence 分数
- **[风险]** 凌晨补充余量可能遗漏事件（若服务在 23:59~00:01 不可用） → **缓解**: 定时任务重试 3 次间隔 60s；补充失败时日志告警，手动触发可补
- **[权衡]** 分步 workflow 增加 DeepSeek API 调用次数（每次日报 3 次调用） → 总计约 3×800 tokens/call ≈ 2400 tokens/日报，DeepSeek 定价极低（$0.14/1M input），成本可忽略

## Migration Plan

1. 新增 migration：`daily_reports` + `weekly_reports` 表
2. 部署新代码（向后兼容——现有 API 端点保留，新增持久化端点）
3. 首次启动时 seed 填充 report 状态
4. 无需回滚：旧表和数据不受影响
